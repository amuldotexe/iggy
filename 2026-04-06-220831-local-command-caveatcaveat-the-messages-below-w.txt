⏺ Iggy Sink Connector Architecture -- ELI5 with ASCII Art

  LAYER 0: THE ESSENCE (read this and you get 80% of it)

  A sink connector is a waiter who picks up dishes (messages) from the Iggy 
  kitchen, cleans them up, and delivers them to whatever restaurant table 
  (database/service) you choose.

  The entire plugin contract is just 3 methods. That is it.

      ┌─────────────────────────────────────────────┐
      │           THE SINK TRAIT (3 methods)         │
      │                                              │
      │   open()    -- Seat yourself, get ready      │
      │   consume() -- Take the plate, serve it      │
      │   close()   -- Clean up, go home             │
      │                                              │
      └─────────────────────────────────────────────┘

      ┌──────────┐    poll     ┌──────────┐  transform  ┌──────────┐
      │  Iggy    │───batches──>│ Runtime  │────clean────>│  YOUR    │
      │  Server  │   of msgs   │  Engine  │   & shape    │  SINK    │
      │          │◀──TCP/QUIC──│          │──postcard───>│ (plugin) │
      └──────────┘             └──────────┘   via FFI    └──────────┘

  The actual Rust trait:

   #[async_trait]
   trait Sink: Send + Sync {

       async fn open(&mut self) -> Result<(), Error>;
       //            ^^^^^^^^
       //            &mut = you can store state (connection pools, etc.)

       async fn consume(
           &self,                               // shared ref: multiple tasks OK
           topic_metadata: &TopicMetadata,      // which stream + topic
           messages_metadata: MessagesMetadata, // partition, offset, schema
           messages: Vec<ConsumedMessage>,       // the actual data!
       ) -> Result<(), Error>;

       async fn close(&mut self) -> Result<(), Error>;
   }

  Key insight: open and close take &mut self (exclusive access), but consume 
  takes &self (shared). The runtime can call consume concurrently after setup.

⏺ The Sink Trait: A Masterclass in Rust Concurrency Design

  ---
  Layer 0 -- ELI5: The Theater Metaphor

  ONE stagehand sets up the stage alone. Once the curtain rises, MANY actors 
  perform simultaneously. ONE stagehand tears it down alone.

      ACT I: SETUP              ACT II: THE SHOW           ACT III: TEARDOWN
      (&mut self)                (&self)                    (&mut self)

     ONE stagehand             MANY actors at once         ONE stagehand
     sets up alone             share the stage             cleans up alone

    ┌─────────────┐     ┌─────────────────────────┐     ┌─────────────┐
    │  ┌───────┐  │     │  O    O    O    O    O  │     │  ┌───────┐  │
    │  │ open()│  │     │ /|\  /|\  /|\  /|\  /|\ │     │  │close()│  │
    │  │       │  │     │ / \  / \  / \  / \  / \ │     │  │       │  │
    │  │  ONE  │  │     │                         │     │  │  ONE  │  │
    │  │PERSON │  │     │   consume() x N         │     │  │PERSON │  │
    │  └───────┘  │     │   ALL AT THE SAME TIME  │     │  └───────┘  │
    └─────────────┘     └─────────────────────────┘     └─────────────┘
          |                        |                          |
     "No one else          "Everyone can read           "No one else
      touch anything!"      the script together"         touch anything!"

  ---
  Layer 1 -- The Borrow Checker's View

  Rust's fundamental law: either ONE writer or MANY readers. Never both.

    TIMELINE OF BORROW STATES

    Time ──────────────────────────────────────────────────────▶

    Phase:   OPEN            CONSUME CONSUME CONSUME      CLOSE
             (&mut self)     (&self) (&self) (&self)      (&mut self)

             ┌─────┐
    &mut ────┤LOCK │──────────────────────────────────────┌─────┐
             └─────┘                                      │LOCK │
                                                          └─────┘
                       ┌────┐ ┌────┐ ┌────┐
    &self ─────────────┤ ok ├─┤ ok ├─┤ ok ├───────────────────────
                       └────┘ └────┘ └────┘
                       ▲             ▲
                       └─ these can OVERLAP ─┘

    METHOD-BY-METHOD BREAKDOWN

    ┌─────────────────────────────────────────────────────┐
    │  async fn open(&mut self)                           │
    │  "I need EXCLUSIVE access to set things up"         │
    │  - Initialize connection pools                      │
    │  - Write to self.connection = Some(conn)            │
    │  Borrow: &mut self  =  ONE caller, NO concurrency  │
    └─────────────────────────────────────────────────────┘
                            │
                            ▼
    ┌─────────────────────────────────────────────────────┐
    │  async fn consume(&self, ...)                       │
    │  "I only need to READ my config and push data"      │
    │  - Read from self.pool (already initialized)        │
    │  - Any mutation goes through interior mutability    │
    │  Borrow: &self  =  MANY callers, FULL concurrency  │
    └─────────────────────────────────────────────────────┘
                            │
                            ▼
    ┌─────────────────────────────────────────────────────┐
    │  async fn close(&mut self)                          │
    │  "I need EXCLUSIVE access to tear things down"      │
    │  - Flush buffers, close connections                 │
    │  Borrow: &mut self  =  ONE caller, NO concurrency  │
    └─────────────────────────────────────────────────────┘

    trait Sink: Send + Sync
                ────   ────
                 │       │
                 ▼       ▼
    ┌──────────────────────────┐   ┌──────────────────────────┐
    │  Send enables:           │   │  Sync enables:           │
    │                          │   │                          │
    │  tokio::spawn(async {    │   │  task1: sink.consume()   │
    │    sink.open().await     │   │  task2: sink.consume()   │
    │  })                      │   │  task3: sink.consume()   │
    │                          │   │                          │
    │  Sink MOVED across       │   │  &self SHARED across     │
    │  threads                 │   │  threads simultaneously  │
    └──────────────────────────┘   └──────────────────────────┘


    What breaks without Sync:

    struct BrokenSink {
        cache: RefCell<HashMap<...>>,   // RefCell is !Sync
    }

    Thread A        Thread B
       │               │
       ▼               ▼
    borrow_mut()    borrow_mut()   <── BOTH at once!
       │               │                RACE CONDITION!
       └───────┬───────┘                RefCell uses a plain
               ▼                        integer, not an atomic
         ╳ COMPILE ERROR

    Fix:  RefCell<T>  ──▶  Mutex<T>    (Sync-safe)
          Cell<T>     ──▶  AtomicU64   (Sync-safe)
          Rc<T>       ──▶  Arc<T>      (Sync-safe)

  The One-Line Summary

  ▎ The &mut self / &self split encodes the lifecycle state machine into the 
  type system -- setup and teardown are single-threaded by compile-time 
  guarantee, while the hot path (consume) scales to N concurrent tasks with zero
   external locking. This is Rust's borrow checker turning concurrency bugs into
   compile errors.

  PART 7: The &mut vs & Trick -- WHY It Matters

  The Whiteboard Analogy

  Setup (&mut self = exclusive): Someone is DRAWING columns on a whiteboard.
  Everyone else stays back.

    ┌────────────────────────────────┐
    │          WHITEBOARD            │
    │     O  "I'm drawing.          │
    │    /|\ Everyone STAY BACK."   │
    │    / \                        │
    │         X  X  X  <-- waiting  │
    └────────────────────────────────┘

  Work (&self = shared): Whiteboard is ready. EVERYONE reads it at the same
  time.

    ┌────────────────────────────────┐
    │          WHITEBOARD            │
    │  ┌──────┬──────┬──────┐       │
    │  │ Name │ Task │ Done │       │
    │  └──────┴──────┴──────┘       │
    │  O  O  O  O  O  <-- everyone  │
    │  reading at the same time     │
    └────────────────────────────────┘

  Cleanup (&mut self = exclusive): One person erases. Everyone else stays back.

  WHY this makes things fast:

    WITH &self (shared -- what Iggy uses):

    Batch 1: ████████████████
    Batch 2:    ████████████████
    Batch 3:       ████████████████
                                     Total: FAST!

    WITHOUT (exclusive -- if Iggy did it wrong):

    Batch 1: ████████████████
    Batch 2:                 ████████████████
    Batch 3:                                 ████████████████
                                     Total: 3x SLOWER!

  99.9% of the work (building SQL, executing queries, printing) is lock-free.
  Only the tiny counter needs a lock for microseconds.

    ┌──────────────────────────────────────┐
    │  PostgresSink with &self (shared)    │
    │                                      │
    │  pool, config ── read-only, shared   │
    │  (99.9% of the work happens here)    │
    │                                      │
    │  counter ── locked for microseconds  │
    │  (0.1% of the work)                  │
    │                                      │
    │  = FAST PARALLEL PROCESSING          │
    └──────────────────────────────────────┘

⏺ Rust's self Variations -- The Complete Guide (Zero Rust Knowledge Required)

  The One Metaphor: YOUR BOOK

  Every variation of self is about what happens to your book when you hand it to
   someone.

  ---
  Variation 1: self -- "Take my book. It's yours now."

  You hand your book to someone. They keep it. You no longer have a book.

    BEFORE                    DURING                   AFTER

    ┌─────┐                 ┌─────┐
    │ YOU │  "Here, take   │THEM │  "Thanks,       ┌─────┐
    │     │   my book."    │     │   I'll keep it" │ YOU │
    │ [📖]│───────────────>│ [📖]│                 │     │
    └─────┘                └─────┘                 │ [ ] │ EMPTY!
                                                    └─────┘
    You HAD a book.        They USE it.            You have NOTHING.
                                                    Touch it? ERROR.

  fn destroy(self) { ... }
  //         ^^^^
  // "self" with no & means: give me the whole thing.
  // After this call, the original is GONE.

  Keyword by keyword:
  - fn = function (a named action)
  - self = "the object itself" -- not a reference, the ACTUAL thing
  - After calling this, the object is consumed. Dead. Gone.

  When to use: Conversion functions ("turn yourself into something else"),
  one-shot operations ("fire and forget"), builder patterns ("finalize and
  return the result").

  When NOT to use: When you need the object again later.

  ---
  Variation 2: mut self -- "Take my book AND scribble in it. It's yours."

  Same as self, but the receiver is allowed to modify it before consuming it.

    BEFORE                    DURING                   AFTER

    ┌─────┐                 ┌─────┐
    │ YOU │  "Here, take   │THEM │  "I'll write    ┌─────┐
    │     │   my book."    │     │   in it, then   │ YOU │
    │ [📖]│───────────────>│ [📖✏️]│  keep it."     │     │
    └─────┘                └─────┘                 │ [ ] │ EMPTY!
                                                    └─────┘

  fn consume_and_modify(mut self) { ... }
  //                    ^^^^^^^^
  // "mut" = I'm allowed to change it
  // "self" = I own it now (original is gone)

  Keyword by keyword:
  - mut = "mutable" = "changeable" = I can modify this
  - self = the actual object (not a reference)

  When to use: Builder pattern -- you chain modifications then finalize:
  Button::new()          // create it       (no self)
      .label("Click me") // modify + return (mut self)
      .color("red")      // modify + return (mut self)
      .build()           // finalize        (self -- consumed)

  When NOT to use: When you still need the object after the call.

  ---
  Variation 3: &self -- "Read my book over my shoulder. MANY people can read at 
  once."

  You let someone look at your book. You still own it. They cannot write in it.
  Multiple people can read simultaneously.

    BEFORE                    DURING                   AFTER

    ┌─────┐                 ┌─────┐                 ┌─────┐
    │ YOU │  "Look, but    │ YOU │  O O O          │ YOU │
    │     │   don't touch" │     │  │ │ │ reading  │     │
    │ [📖]│               │ [📖]│◄─┘ │ │          │ [📖]│
    └─────┘               └─────┘    │ │          └─────┘
                            Still     │ │
                            YOURS!    └─┘ Many     STILL yours!
                                      can look     Nothing changed.

  fn read_title(&self) -> String { ... }
  //            ^^^^^
  // & = "reference" = I'm just LOOKING, not taking
  // self = at myself
  // Combined: "a shared, read-only peek at myself"

  Keyword by keyword:
  - & = "reference" = a pointer/peek, NOT the actual thing
  - self = the object
  - &self together = "let me look at myself without taking ownership"

  When to use: Reading data, computing values, checking state. Anything that
  does NOT change the object. This is the most common one.

  When NOT to use: When you need to modify the object's fields.

  ---
  Variation 4: &mut self -- "Lend my book to ONE person. They can write in it. 
  They MUST give it back."

  You lend your book to exactly ONE person. They can write notes in it. But they
   MUST return it. Nobody else can touch it while they have it.

    BEFORE                    DURING                   AFTER

    ┌─────┐                 ┌─────┐                 ┌─────┐
    │ YOU │  "Here, you    │THEM │  "I'm writing   │ YOU │
    │     │   can write    │     │   in it. Nobody  │     │
    │ [📖]│──────lend─────>│ [📖✏️]│  else touch!"   │ [📖]│
    └─────┘                └─────┘                 └─────┘
    You WAIT.              ONE person.              Book is BACK!
    Can't use it.          Exclusive.               (with changes)

  fn set_name(&mut self, name: String) { ... }
  //          ^^^^^^^^^
  // & = reference (borrowing, not owning)
  // mut = mutable (can change things)
  // self = the object
  // Combined: "lend me exclusive, writable access to myself"

  Keyword by keyword:
  - & = reference (borrowing -- will give it back)
  - mut = mutable (allowed to change things)
  - self = the object
  - &mut self together = "exclusive, writable borrow that gets returned"

  When to use: When you need to modify the object AND still use it afterward.
  Setup methods, configuration, state changes.

  When NOT to use: When you only need to read. Use &self instead -- it allows
  parallelism.

  ---
  Variation 5: No self at all -- "I don't need a book. I AM the bookstore."

  This is a function that belongs to the TYPE itself, not to any particular
  instance. Like asking "How many books does the BookStore chain have?" vs "How
  many pages does THIS book have?"

    NO BOOK NEEDED

    ┌────────────────────┐
    │   BOOKSTORE (type) │
    │                    │
    │   BookStore::new() │  "I CREATE books.
    │                    │   I don't NEED one
    │   BookStore::count()│   to exist first."
    └────────────────────┘

  fn new(name: String) -> Self { ... }
  // No self parameter at all!
  // Called as: BookStore::new("My Store")
  // Not as: my_store.new(...)

  When to use: Constructors (new), factory methods, utility functions that
  belong to a type but don't need an instance.

  ---
  Variation 6: Smart Pointers -- "The book is in a special case"

  These are advanced. The book is wrapped in a container that provides extra
  abilities.

    Box<Self>     = Book in a locked box. Only ONE owner.
                    Just like self, but the book is on a
                    different shelf (heap memory).

    Rc<Self>      = Book with a sign-out sheet. MULTIPLE
                    owners, but only on ONE thread.
                    When last person returns it, book is destroyed.

    Arc<Self>     = Same as Rc, but the sign-out sheet is
                    THREAD-SAFE. Multiple threads can own it.

    Pin<&mut Self> = Book NAILED to the table. You can write
                     in it, but you CANNOT move it. Used for
                     async/await internals.

    ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ Box<Self>│  │ Rc<Self> │  │Arc<Self> │  │Pin<&mut>│
    │          │  │          │  │          │  │          │
    │  ┌────┐  │  │  ┌────┐  │  │  ┌────┐  │  │  ┌────┐  │
    │  │ 📖 │  │  │  │ 📖 │  │  │  │ 📖 │  │  │  │📌📖│  │
    │  └────┘  │  │  └────┘  │  │  └────┘  │  │  └────┘  │
    │ 1 owner  │  │ N owners │  │ N owners │  │ CAN'T   │
    │ on heap  │  │ 1 thread │  │ N threads│  │ MOVE IT │
    └──────────┘  └──────────┘  └──────────┘  └──────────┘

                               Can't consume or close later!

                               &self can't write the new
                               database connection into itself.

                               &mut self: modify + keep alive. ✓

    consume()    &self         &mut self would force ONE batch
                               at a time. SLOW. Like a single-
                               lane highway.

                               &self allows MANY batches in
                               parallel. FAST. Multi-lane. ✓

                               immediately. The container
                               couldn't inspect it or report
                               status afterward.

                               &self can't set connection=None
                               or flush buffers (those are writes).

                               &mut self: clean up + let container
                               handle the rest. ✓

  COMMON MISTAKES (In Plain English)

  Mistake 1: Using the object after self consumed it
    sink.destroy();    // self -- sink is consumed
    sink.read();       // ERROR: "value used after move"
                       //        "You gave it away! It's gone!"

  Mistake 2: Trying to write in a &self method
    fn read_only(&self) {
        self.count = 5;  // ERROR: "cannot assign to field
                         //        behind a & reference"
                         // "You said read-only. Writing is cheating."
    }

  Mistake 3: Two &mut self borrows at the same time
    let a = &mut book;
    let b = &mut book;  // ERROR: "cannot borrow as mutable
                         //        more than once"
                         // "You lent the book to A. You can't
                         //  also lend it to B simultaneously."

  Mistake 4: &self and &mut self at the same time
    let reader = &book;
    let writer = &mut book;  // ERROR: "cannot borrow as mutable
                              //        because it is also borrowed
                              //        as immutable"
                              // "Someone is reading! You can't let
