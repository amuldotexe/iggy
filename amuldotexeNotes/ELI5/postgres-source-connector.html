How Iggy's Postgres Source Connector Actually Works

A guided tour for someone who does not know Rust, Apache Iggy, or why state files matter.

Scroll to explore
Act I

The Problem

This repository currently ships a Postgres source connector, which means the data direction is Postgres -> Iggy. Its job is simple to say and tricky to do well: notice new rows or new database changes, package them as Iggy messages, and resume correctly after restarts.

01

A Clerk, Not A Warehouse

The connector is not a full database sync engine. It behaves more like a clerk who checks what is new, labels each package, and hands the batch to the shipping desk.

The three actors that matter are: PostgreSQL as the source of truth, the Postgres connector crate as the reader and packager, and the connectors runtime as the part that actually sends messages into Iggy and saves the checkpoint file.

If you do not know Rust, the core lifecycle words are pleasantly ordinary: open means start up, poll means check for new work, and close means shut down. That is the whole heartbeat we are about to trace.

Working Thesis

The Postgres source connector reads from Postgres and builds batches, but the shared runtime owns the final delivery into Iggy and the durable state-file write.

Act II

The Core Loop

The connector's real rhythm is open -> wait -> read -> package -> publish -> checkpoint. The important twist is that the last two steps happen in the runtime, not inside the Postgres-specific crate.

1

Open

The runtime loads the connector as a dynamic library, passes in JSON config plus any saved state, and calls open(). The connector creates a Postgres connection pool, runs SELECT 1 as a smoke test, and, in CDC mode, may create a publication and replication slot.

2

Sleep, Then Poll

Every poll() call begins with sleep(poll_interval), so the first fetch happens after one full interval. Then the connector picks one branch: table polling or CDC.

3

Build Messages

Rows or change records become ProducedMessage values. The connector chooses a schema of json, raw, or text, stamps IDs and timestamps, and optionally returns a serialized state snapshot alongside the batch.

4

Runtime Publishes To Iggy

The SDK serializes the batch over the plugin boundary, the runtime decodes it, applies any configured transforms, re-encodes it for the target stream schema, and sends it to the configured Iggy stream and topic.

5

Save Progress

Only after a successful send does the runtime write the connector state file. That saved state becomes the resume point for the next process start.

Key Insight

The Postgres crate never calls an Iggy client directly. It hands batches back to the shared runtime, which is why publishing, transforms, metrics, and durable state saving all live outside the connector itself.

Act III

Key Variants Or Modes

The connector has two very different personalities. Polling asks tables for rows. CDC asks Postgres for change records from a logical replication slot.

Polling Mode

  • Builds a SELECT * FROM table ... ORDER BY tracking_column ASC LIMIT batch_size query, unless you supply a custom query.
  • Tracks progress per table in a local tracking_offsets map stored inside connector state.
  • Can start from a saved offset, an initial_offset, or the beginning if neither exists.
  • Supports queue-like tables by deleting rows after reading or marking them processed with a boolean column.

CDC Mode

  • Reads from pg_logical_slot_get_changes(...) instead of repeatedly scanning application tables.
  • Can auto-create publication and slot only when enable_wal_cdc = true; otherwise it assumes those objects already exist.
  • Parses textual logical decoding messages into INSERT, UPDATE, and DELETE records.
  • The documented pg_replicate backend is not implemented in this crate today unless future feature work lands; the working path here is the builtin backend.
payload

Full Row Envelope

Without payload_column, the connector wraps data in a DatabaseRecord shape with table name, operation type, timestamp, row data, and optional old data.

payload_column

Direct Payload Extraction

If you point at one column, the connector can emit that column directly as raw bytes, UTF-8 text, or JSON bytes. This is the cleanest way to avoid the row wrapper.

custom_query

Bring Your Own SQL

The connector can replace placeholders like $table, $offset, $limit, and $now. That gives flexibility, but you own the correctness of the SQL text.

Representative snippet
poll():
  sleep(poll_interval)
  if mode == "polling":
    SELECT * FROM "table"
    WHERE "tracking_column" > last_offset
    ORDER BY "tracking_column" ASC
    LIMIT batch_size
  else if mode == "cdc":
    SELECT lsn, xid, data
    FROM pg_logical_slot_get_changes(slot_name, NULL, NULL, ...)
Act IV

State, Safety, And Recovery

This is where most connector explanations become fuzzy, so let's be precise. The crate keeps some state in memory, serializes it into a blob, and the runtime writes that blob to disk only after successful publish to Iggy.

poll Postgres | v build ProducedMessages + optional state blob | +--> runtime sends to Iggy successfully | | | +--> save source_KEY.state | +--> runtime send fails | +--> do not advance local state file
state

What Is Persisted

The state blob contains last_poll_time, a per-table tracking_offsets map, and processed_rows. In polling mode, tracking_offsets are the real resume markers. In CDC mode, the deeper progress marker mostly lives in PostgreSQL's replication slot.

success

What Happens On Success

If the runtime publishes the batch successfully, it saves the returned state file. On restart, the connector restores that file and continues from the remembered offsets instead of scanning from scratch.

failure

What Happens On Failure

Transient polling reads retry with backoff. If publish to Iggy fails, the local state file is not updated, so plain offset-based polling can replay rows next time. But cleanup modes complicate that story because source rows may already be deleted or marked processed before publish is confirmed.

Reliability Note

The safest replay model here is polling with a stable tracking column and no source-side cleanup. The riskiest modes are delete_after_read, processed_column, and CDC slot consumption, because those advance or mutate the Postgres-side notion of progress before the runtime has finished confirming delivery into Iggy.

Act V

Architecture And Boundaries

The easiest way to understand the architecture is to separate what the connector owns from what the runtime owns. The boundary is much cleaner than it first appears.

connectors runtime | +--> SDK / FFI wrapper | +--> postgres_source plugin | | | +--> open / poll / close | +--> Postgres pool or replication slot | +--> decode / transform / encode / publish +--> save state file +--> metrics, status, start/stop/restart

What The Connector Owns

  • Open the Postgres connection pool and optional CDC setup.
  • Build polling SQL or read logical change records.
  • Turn rows into JSON, text, or raw payloads.
  • Keep short-lived in-memory state and serialize it when returning a batch.

What The Runtime Owns

  • Load the plugin, invoke open, start the background polling loop, and call close.
  • Decode payloads, apply configured transforms, encode for the target topic, and publish into Iggy.
  • Persist state files, expose metrics and HTTP status, and manage restart or shutdown.
Practical Translation

When you read the Rust files, treat the Postgres crate as the adapter and the runtime as the control plane. That mental split makes the codebase much easier to navigate.

Act VI

Failure Modes And Tradeoffs

The connector is refreshingly explicit about some failures and a little optimistic about others. The right mental model is not "perfectly exactly once," but "understand where replay is possible and where progress can outrun delivery."

Retryable

  • Transient SQLx errors during polling reads, including I/O issues and pool timeouts.
  • Selected PostgreSQL database error codes such as serialization failures and deadlocks.
  • Backoff grows by retry_delay * attempt_number, so later retries wait longer.

Fatal

  • Bad mode values, bad identifiers, or startup failures such as "cannot connect to Postgres".
  • CDC requirements not met, such as missing logical WAL setup or unavailable backend support.
  • Publish failures to Iggy, invalid payload extraction, or incorrect custom SQL.
tradeoff

Correctness

Plain polling without cleanup leans toward replay over silent loss. Cleanup options make the source table tidier, but they can erase replay opportunities if the later Iggy send fails.

tradeoff

Performance

batch_size, poll_interval, and max_connections control how aggressively the connector reads. One easy-to-miss detail: startup is not eager, because the first real poll happens after one sleep interval.

tradeoff

Operability

The connector redacts the connection string in logs, can switch to more verbose info-level logging, and benefits from runtime-level metrics and on-disk state files when you need to debug restart behavior.

Skeptical Lens

If you want the cleanest correctness story, avoid assuming that every mode has the same delivery semantics. Polling with offsets, polling with source cleanup, and CDC via slot consumption behave differently under failure.

Act VII

The Big Picture

Here is the short version you can keep in your head after closing the browser: this connector is a Postgres reader plus message packager. The runtime is the publisher plus checkpoint clerk.

Dimension Polling CDC Why It Matters
Input Table rows returned by SQL queries Logical change records returned by a replication slot You choose between scanning for rows and subscribing to database changes.
Progress Anchor Local tracking_offsets in a state file Replication slot progress in Postgres, plus local stats state Recovery depends on different systems in each mode.
Payload Shape Full row envelope or one extracted payload column Change-event envelope for INSERT, UPDATE, DELETE Consumers may receive snapshots of rows or a stream of change events.
Failure Posture Replay-friendly if you only advance local offsets Slot consumption can outrun downstream confirmation The recovery story is configuration-sensitive, not universal.
One Line Summary

Connect to Postgres, wait, read what is new, hand the batch to the runtime, publish to Iggy, and only then trust the checkpoint you can safely resume from.
