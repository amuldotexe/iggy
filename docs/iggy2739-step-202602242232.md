## Known Risk: Telemetry-Only Mismatch Detection

**Impact**: MEDIUM (observability gap, not silent state corruption)

**Current behavior**:
- Mismatch detection is logged via warnings (complete and partial)
- Processing flow does not stop on partial mismatch; warnings are diagnostic only

**Edge cases**:

- State path wiped or connector ID changed
- Multiple source instances without shared state
- Custom ObjectId tracking field not named `_id`

**Mitigation**:
- Complete + partial mismatch warnings in source mark/delete paths
- ObjectId mark test verifies `processed=true` and `processed=false == 0`

## Pre-PR Hardening Spec

### 1. Fix Checkpoint Ordering (Match Postgres Pattern)

**Problem**: MongoDB source updates offset BEFORE mark/delete at lib.rs:455-469. If mark/delete fails, offset is checkpointed but documents aren't processed.

**Postgres pattern** (correct): mark/delete first at lib.rs:560-576, then update state.

**Fix**: Reorder to:

1. Mark/delete FIRST
2. THEN update state (only if mark/delete succeeded)

**Files**: `core/connectors/sources/mongodb_source/src/lib.rs:455-469`

## REGRESSION FOUND → ALL FIXED ✅

### P0: Stale Offset Bug → FIXED

**What was done wrong**: The checkpoint ordering fix moved mark/delete BEFORE state update, BUT mark/delete still reads offset from `self.state` (persisted), not from current batch's `max_offset`.

**Fix applied**: Pass `max_offset.as_deref()` directly to `delete_processed_documents()` and `mark_documents_processed()` as `current_offset` parameter instead of reading from state.

**Commit**: `6aa135c2`

---

### P1: Observability Warnings Too Coarse → FIXED

**What was done wrong**: Warnings trigger on `deleted_count == 0` without comparing to expected batch size.

**Fix applied**: Pass `expected_count` (messages.len()) to mark/delete, only warn if `expected > 0 && actual == 0`.

**Commit**: `c5d8ee3e`

---

### P2: Sink README Overstates Dedup → FIXED

**What was done wrong**: README says `_id` is used "for deduplication" but sink is insert-only and duplicate `_id` is hard failure, not graceful dedup.

**Fix applied**: Changed wording from "for deduplication" to "for document identity" and clarified "Insert-only mode: duplicate key error is a hard failure".

**Commit**: `c5d8ee3e`

### Lessons Learned

1. **Code review gap**: When changing operation order, must trace the DATA flow through to all consumers
2. **Test gap**: Tests verify end state, not the specific offset VALUE used in mark/delete operations
3. **Integration vs unit**: Unit tests mock state, E2E tests verify final state - neither catches "which offset is used"

## Improvement Spec: Partial Mismatch Telemetry (Implemented)

### Background

Before `3f8f59fb`, warnings only fired when `expected > 0 && actual == 0` (complete mismatch).
Partial mismatch (`actual < expected`) was silent and could indicate:

- Filter not matching all expected documents (state drift)
- Some documents already processed/missing
- Race conditions in concurrent access

### Precedent Analysis (CORRECTED)

- Elasticsearch sink: partial success accounting YES (`errors` vs `documents_indexed`), warning on partial NO, location `lib.rs:227, 241`
- MongoDB sink: partial success accounting YES (`successful_inserts` vs `insertion_errors`), warning on partial NO, location `lib.rs:222, 247`
- Postgres source: partial success accounting NO, warning on partial NO, location `lib.rs:615, 644`
- **MongoDB source**: partial success accounting YES (`expected_count`), warning on partial YES, location `lib.rs:635, 687`

**Key insight**: Sinks track partial success as accounting, while MongoDB source now emits explicit warning telemetry for both partial and complete mismatches.

**Pattern**:

- Sinks: partial success accounting → metrics, NO warnings
- Sources: MongoDB now has complete + partial mismatch warnings

### Decision

**Status**: IMPLEMENTED

**Rationale**:

- Sinks already track partial success in state (accounting precedent exists)
- MongoDB source already had the infrastructure (`expected_count` + operation result counts)
- Logging-only change, no behavior/control-flow change

**Implementation**:

```rust
match classify_expected_actual_mismatch(expected_count, actual_count) {
    ExpectedActualCountMismatch::None => debug!(...),
    ExpectedActualCountMismatch::Partial { expected, actual } => warn!(expected, actual, ...),
    ExpectedActualCountMismatch::Complete { expected } => warn!(expected, actual = 0, ...),
}
```

**Noise mitigation**:

- No warning when `expected_count == 0` (idle path)
- No warning when `actual == expected` (normal path)
- Complete mismatch (`actual == 0`) is still warned explicitly with separate message

### Risk

| Risk                                | Likelihood | Mitigation                                   |
| ----------------------------------- | ---------- | -------------------------------------------- |
| False positive on concurrent access | Medium     | Acceptable - indicates race condition exists |
| Log noise in high-throughput        | Low        | Only triggers on actual mismatch             |
