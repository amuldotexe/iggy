## Known Risk: Silent Mark Failure

**Impact**: HIGH in wrong conditions (state reset/corruption/redeploy = duplicate replay flood)

**Why silent**: `update_many` result not validated for matched/modified count; `0 matched` treated as success

**Edge cases**:
- State path wiped or connector ID changed
- Multiple source instances without shared state
- Custom ObjectId tracking field not named `_id`

**Mitigation**: Test now verifies `processed=true` was set

## Pre-PR Hardening Spec

### 1. Fix Checkpoint Ordering (Match Postgres Pattern)

**Problem**: MongoDB source updates offset BEFORE mark/delete at lib.rs:455-469. If mark/delete fails, offset is checkpointed but documents aren't processed.

**Postgres pattern** (correct): mark/delete first at lib.rs:560-576, then update state.

**Fix**: Reorder to:
1. Mark/delete FIRST
2. THEN update state (only if mark/delete succeeded)

**Files**: `core/connectors/sources/mongodb_source/src/lib.rs:455-469`

---

### 2. Add Observable Warnings for Mark/Delete

**Problem**: `update_many` / `delete_many` return counts but we ignore them. Silent 0-match treated as success.

**Fix**: Pass `expected_count` to helper functions, log warning when counts suspicious:
```rust
if result.deleted_count == 0 && expected_count > 0 {
    tracing::warn!(expected = expected_count, deleted = result.deleted_count, ...);
}
```

**Files**: `core/connectors/sources/mongodb_source/src/lib.rs:589`, `lib.rs:612`

---

### 3. Strengthen ObjectId Mark Test

**Files**: `core/integration/tests/connectors/mongodb/mongodb_source.rs:715-734`

## REGRESSION FOUND → ALL FIXED ✅

### P0: Stale Offset Bug → FIXED

**What was done wrong**: The checkpoint ordering fix moved mark/delete BEFORE state update, BUT mark/delete still reads offset from `self.state` (persisted), not from current batch's `max_offset`.

**Fix applied**: Pass `max_offset.as_deref()` directly to `delete_processed_documents()` and `mark_documents_processed()` as `current_offset` parameter instead of reading from state.

**Commit**: `6aa135c2`

---

### P1: Observability Warnings Too Coarse → FIXED

**What was done wrong**: Warnings trigger on `deleted_count == 0` without comparing to expected batch size.

**Fix applied**: Pass `expected_count` (messages.len()) to mark/delete, only warn if `expected > 0 && actual == 0`.

**Commit**: `c5d8ee3e`

---

### P2: Sink README Overstates Dedup → FIXED

**What was done wrong**: README says `_id` is used "for deduplication" but sink is insert-only and duplicate `_id` is hard failure, not graceful dedup.

**Fix applied**: Changed wording from "for deduplication" to "for document identity" and clarified "Insert-only mode: duplicate key error is a hard failure".

**Commit**: `c5d8ee3e`

---

### Lessons Learned

1. **Code review gap**: When changing operation order, must trace the DATA flow through to all consumers
2. **Test gap**: Tests verify end state, not the specific offset VALUE used in mark/delete operations
3. **Integration vs unit**: Unit tests mock state, E2E tests verify final state - neither catches "which offset is used"
