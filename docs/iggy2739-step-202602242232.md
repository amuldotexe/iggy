## Known Risk: Silent Mark Failure

**Impact**: Can be **HIGH** in wrong conditions.

### Risk Pattern

1. **Steady state (state file healthy)**: Medium/low immediate impact
   - If marking silently fails, offset advances first (line 456)
   - Won't usually re-read old docs in same run
   - Can hide for a while

2. **State reset/corruption/redeploy**: HIGH IMPACT (duplicate replay flood)
   - If state cannot deserialize, connector falls back to empty/initial state (line 186, sdk line 66)
   - If documents were never marked, restart can replay large historical sets

3. **Why this can be silent**
   - `update_many` result is NOT validated for matched/modified count
   - `0 matched` is still treated as success (line 626)
   - Wrong filter/type/field can no-op without error

### Concrete Edge Cases That Trigger It

- State path wiped, corrupted, or connector ID changed
- Running multiple source instances without shared consistent state
- Regression in mark filter/operator/field name
- `processed_field` docs not actually boolean `false`
- Custom ObjectId tracking field not named `_id` (conversion only handles `_id`) (line 115)

### Type Mismatch Severity

MongoDB comparison behavior verified:

- String vs ObjectId comparisons return NO matches
- `gt_string=0`, `gt_oid=2`
- `lte_string_del=0`, `lte_oid_del=2`
- One bad type path makes mark/delete effectively no-op

### Test Gap

The ObjectId mark test does NOT verify `processed=true` due to fixture limitation (line 715).
Mark-path regressions can slip through even when all tests are green.

## Test Gap Fix: ObjectId Mark Verification

### Problem Statement

The test `source_mark_processed_with_object_id` does NOT verify that `processed=true` was actually set in MongoDB. Lines 715-717 admit this limitation:

```rust
// Note: We can't easily count "processed=true" documents because the fixture's
// count_processed_documents uses "is_processed" field, but our fixture uses "processed"
// The key verification is that documents weren't deleted and second batch was polled correctly
```

### Why This Matters

- **Risk**: Silent mark failure can cause duplicate replay floods on state reset
- **Type mismatch** (string vs ObjectId) makes mark/delete effectively no-op
- **Current test gap** means mark-path regressions could slip through

### Precedent in Codebase

1. **Postgres source test** (`postgres_source.rs:339-402`) verifies before/after counts:
   - `initial_unprocessed = fixture.count_unprocessed(&pool)`
   - `initial_processed = fixture.count_processed(&pool)`
   - Asserts `final_processed == TEST_MESSAGE_COUNT`

2. **MongoDB non-ObjectId mark test** (`mongodb_source.rs:243-258`) verifies:
   - `count_processed_documents(&mongo_client)` returns expected count

### Fix Specification

#### Step 1: Fixture Already Has the Method

The `count_documents_by_field<T>()` method exists at `source.rs:88-103`:

```rust
pub async fn count_documents_by_field<T: serde::Serialize>(
    &self,
    client: &Client,
    field: &str,
    value: T,
) -> Result<u64, TestBinaryError>
```

This generic method can count `processed=true` documents.

#### Step 2: Update ObjectId Mark Test

In `mongodb_source.rs`, function `source_mark_processed_with_object_id`, replace lines 715-717 with:

```rust
// === PHASE 3: Verify ALL documents marked processed=true ===
// Wait for mark to complete on all 6 documents
let mut processed_count = 0u64;
for _ in 0..POLL_ATTEMPTS {
    processed_count = fixture
        .count_documents_by_field(&mongo_client, "processed", true)
        .await
        .unwrap_or(0);
    if processed_count >= 6 {
        break;
    }
    sleep(Duration::from_millis(POLL_INTERVAL_MS)).await;
}

assert_eq!(
    processed_count, 6,
    "Expected all 6 documents to have processed=true, got {processed_count}"
);
```

### Verification

After fix, the test will:

1. Seed 6 documents with `processed: false`
2. Poll all 6 documents (triggers mark)
3. Verify `count_documents_by_field("processed", true) == 6`

This closes the test gap and ensures mark-path regressions are caught.
